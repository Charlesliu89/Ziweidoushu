# 紫微斗数开源依赖与文墨兼容规则

日期：2026-05-30

## 当前生产依赖

本项目当前只把 `iztro` 作为排盘内核。依赖版本以 `package.json` 和 `package-lock.json` 为准；生成的 JSON 必须在 `engine.version` 中记录实际加载的 `iztro` 版本。不得把历史调研笔记、未重新核验的仓库许可或第三方规则描述当作生产依据。

## 规则取舍

底层排盘继续交给 `iztro`，项目自身只负责稳定输入、稳定 JSON 输出和文墨风格标记层：

- 宫名口径：`仆役` 显示为 `交友宫`；本命盘、流限和文墨摘要都必须包含 12 个唯一固定宫名。
- JSON schema：当前输出 `schemaVersion` 为 `wenmo-ziwei-json/v2`，用于标识 `mutagenStars` 等当前字段语义。
- 输入口径：年月日按阳历处理，`input.calendar` 固定为 `solar`。`location` 和 `longitude` 仅保存备查，不参与 `iztro.bySolar` 计算，必须在 JSON 中标记为未用于计算。
- 出生时辰：`input.timeIndex` 必须记录实际传入 `iztro` 的时辰索引；晚子时 23:00-23:59 记录为 `12`。
- 生年四化：显示为 `生年禄/权/科/忌`。
- 宫干自化：本宫宫干四化命中本宫星曜，显示 `↓禄/↓权/↓科/↓忌`，并记录四化飞出目标宫。
- 向心自化：对宫宫干四化命中本宫星曜，显示 `↑禄/↑权/↑科/↑忌`。
- 来因宫：使用 `iztro` 的 `isOriginalPalace` 字段。
- 流限：目标日期存在时，输出大限、流年、流月、流日、流时的 12 宫星曜、12 宫名和 `mutagenStars` 四化星映射；流时必须记录实际使用的目标时辰，未传目标时辰时按 00:00 子时计算并写入 JSON。文墨摘要也应覆盖这些层级。`小限/age` 只保存 `iztro` 提供的基础 scope 字段，不补造星曜明细。
- 原始日期：`natal.rawDates` 只保存结构校验后的农历年月日、闰月标记和年/月/日/时四组干支，不透传未定义结构。
- 干支枚举：四柱、宫干宫支、大限干支和流限干支都必须落在十天干、十二地支枚举内。
- 神煞和四化：每宫必须输出长生十二神、博士十二神、将前十二神、岁前十二神；星曜 `type`、`scope`、`brightness` 必须落在已知枚举内；星曜四化值必须能生成完整 `化禄/化权/化科/化忌` 标签。

## 项目边界

公开仓库只保留可复核的实现和说明：

- `src/cli.ts`：命令行入口，只允许写入项目内本地生成目录中的 `.json` 文件，并可显式传入 `--fix-leap true/false` 和流时目标时分。
- `src/index.ts`：库入口。
- `src/lib/astro.ts`：输入校验、时辰换算和 `iztro` 排盘调用。
- `src/lib/serialize.ts`：把排盘结果序列化为稳定 JSON 结构，并校验原始日期、星曜元数据、小限年龄、大限范围和流限结构。
- `src/lib/wenmo-rules.ts`：生成文墨展示标记和流限摘要。
- `scripts/validate-demo-chart.mjs`：使用演示参数验证核心标记、结构完整性、非法输入和 CLI 边界，不保存真实案例。
- `scripts/privacy-audit.mjs`：上传前检查 `.gitignore` 和公开文件中的隐私风险。
- `docs/`：只放技术说明，不放真实输入、命盘 JSON 或解读记录。

真实输入、排出的盘、导出报告和案例记录必须留在 `.gitignore` 覆盖的本地目录，例如 `output/`、`inputs/`、`private/`、`exports/`。

## 变更准则

- 不新增无法被脚本验证的排盘结论。
- 不复制未核验许可的第三方源码或规则表。
- 不把真实出生资料、客户资料或解读文本提交到公开文件。
- 新增字段时必须能说明来源：来自 `iztro` 原始结果、项目内显式规则，或用户输入原样保存。
- 引擎元数据必须可追溯；`engine.package` 和 `engine.version` 要能对应当前安装依赖。
- 用户输入字段必须保持白名单；未知字段直接拒绝，不透传进生成的 JSON。
- 默认值必须在 JSON 中可见；例如未传 `fixLeap` 时，`input.fixLeap` 仍记录实际生效的 `true`。
